# 블로그 프로젝트 개발 규정 (Gemini CLI 가이드라인)

이 문서는 'commit-to-blog' 프로젝트를 수행할 때 Gemini CLI가 반드시 준수해야 하는 **최우선 지침**입니다.

## 0. 핵심 원칙 (Core Principle)
- **사용자 승인 필수**: 모든 코드 생성, 수정, 파일 생성 및 내용 추가 전, 반드시 사용자에게 계획을 보고하고 **사전 검토와 승인**을 받아야 합니다.

## 1. 참조 저장소 및 문헌 구성 (Reference Organization)
작업 수행 시 참고해야 할 주요 저장소(디렉토리)와 문헌들의 역할은 다음과 같습니다.

*   **`gemini/` (에이전트 전용 가이드 및 기획 자료)**: 에이전트(Gemini CLI)가 이 프로젝트를 어떻게 수행해야 하는지 정의한 절대 가이드와 기본 기획 정보가 담긴 곳입니다.
    *   **`gemini/GEMINI.md`**: 본 파일. 에이전트가 준수해야 할 개발 규정과 소통 원칙이 담긴 최우선 지침서입니다.
    *   **`gemini/plan.md`**: 기능 명세, 기술 스택, 데이터 구조 등 프로젝트의 뼈대가 기술된 마크다운 기획서입니다.
    *   **`gemini/plan.pdf`**: 시각적 레이아웃(와이어프레임) 및 기획의 상세 컨셉이 담긴 문서입니다.
*   **`docs/` (사용자 설계 및 결정 기록)**: 프로젝트 진행 과정에서 사용자가 직접 고민하고 결정한 실제 설계 내용이 기록되는 곳입니다. **에이전트는 구현 시 이곳의 내용을 가장 먼저 확인하여 사용자의 최신 설계 의도를 파악해야 합니다.**
*   **기획 문서 (plan.md, plan.pdf)**: 전체 기능 및 레이아웃의 기준점입니다. 단, 이는 유동적인 가이드이며 필요할 경우 사용자와 논의하여 변경할 수 있습니다.
*   **에이전트 재량 지식**: 위 문헌들에 명시되지 않은 세부적인 코드 구현 방식이나 라이브러리 활용 등은 최신 React/Node.js 기술 관례를 자율적으로 참고합니다.

## 2. 프로젝트 개괄 (Project Overview)
GitHub 커밋 기록을 바탕으로 Gemini AI를 활용해 개발 블로그 포스트를 반자동으로 생성하고 관리하는 **1인용(Personal)** 웹 애플리케이션입니다.

## 3. 수행 방법 및 기술 스택 (Execution Methodology)
- **Framework**: React (Frontend), Node.js / Express.js (Backend)
- **Database**: MongoDB / **AI API**: Gemini API
- **Auth**: 별도 로그인 UI 없이 GitHub Personal Access Token(PAT)을 서버 환경변수(`.env`)로 연동
- **Styling**: 바닐라 CSS 또는 컴포넌트 스코프 스타일링 권장. `plan.html`은 참고용이며 개발 중 더 나은 UI가 있다면 제안하고 컨펌받습니다.

## 4. 주석 작성 가이드라인 (Commenting Guidelines for Learning)
- **기능 단위 주석**: 독립적인 기능을 수행하는 블록 단위로 작성합니다.
- **설계 포인트 및 사유 명시**: 어떤 설계 선택을 했는지(What)와 왜 그렇게 설계했는지(Why)를 상세히 설명합니다.
- **종료 포인트 명시**: `// --- [기능명] 구현 종료 ---` 주석을 활용합니다.
