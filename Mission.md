# 11주차 LLM 블로그 서비스 체크리스트

기준 문서: `11주차 LLM 블로그 서비스 - 스마트블로그 서비스 - GitHub 연동.pdf`

이 문서는 앞으로 AI-Blog 미션을 진행할 때 계속 참고할 기준 체크리스트다.  
새 기능을 만들거나 계획을 수정할 때는 먼저 이 파일을 읽고, 완료한 항목은 체크하거나 근거를 함께 남긴다.

## 1. 미션 개요

- [ ] GitHub 활동 데이터를 분석해 자동으로 개발 블로그를 생성하는 서비스를 만든다.
- [ ] 사용자는 자신의 GitHub 저장소를 선택할 수 있어야 한다.
- [ ] 사용자는 분석할 브랜치와 커밋 로그를 선택할 수 있어야 한다.
- [ ] 선택된 커밋 또는 코드 변경 사항을 LLM이 분석한다.
- [ ] LLM 분석 결과를 개발 블로그 초안 형태로 변환한다.
- [ ] 생성된 초안은 사용자가 직접 수정하고 보완할 수 있어야 한다.
- [ ] 작성 또는 저장된 포스트는 목록에서 다시 확인할 수 있어야 한다.
- [ ] 최종 검토 후 포스트를 발행하는 흐름까지 고려한다.

## 2. 학습 목표

- [ ] 스스로 구체적인 기획과 요구사항을 분석해 개발 항목을 정리한다.
- [ ] AI를 활용한 개발 workflow를 직접 만들고 계획을 수립한다.
- [ ] LLM을 연동한 AI 서비스를 구현한다.
- [ ] GitHub API와 LLM API를 실제 서비스 흐름 안에서 연결한다.
- [ ] 프론트엔드와 서버 역할을 나누어 설계하고 구현한다.
- [ ] 구현 결과를 테스트하고 검증하는 기준을 스스로 세운다.

## 4. 핵심 사용자 흐름

- [ ] 사용자가 서비스에 들어온다.
- [ ] GitHub 저장소 목록을 불러온다.
- [ ] 사용자가 저장소를 선택한다.
- [ ] 선택한 저장소의 브랜치 목록을 불러온다.
- [ ] 사용자가 브랜치를 선택한다.
- [ ] 선택한 브랜치의 커밋 목록을 불러온다.
- [ ] 사용자가 블로그 글의 재료로 사용할 커밋을 선택한다.
- [ ] 선택한 커밋의 메시지, 작성자, 날짜, 변경 파일, diff 또는 요약 정보를 수집한다.
- [ ] 서버가 수집한 GitHub 데이터를 LLM 요청에 적합한 형태로 정리한다.
- [ ] LLM이 개발 과정, 변경 의도, 구현 내용, 배운 점을 블로그 초안으로 작성한다.
- [ ] 사용자는 생성된 초안을 편집기에서 수정한다.
- [ ] 사용자는 포스트를 저장한다.
- [ ] 저장된 포스트는 카드 목록에서 확인한다.
- [ ] 사용자는 저장된 글을 다시 열어 수정한다.
- [ ] 사용자는 최종 검토 후 발행한다.

## 5. 기능 요구사항

### 5.1 블로그 생성 기능

- [ ] GitHub Repository 리스트를 불러온다.
- [ ] 특정 Repository를 선택할 수 있다.
- [ ] 선택한 Repository의 브랜치 목록을 불러온다.
- [ ] 분석할 대상 브랜치를 선택할 수 있다.
- [ ] 선택한 브랜치의 커밋 로그를 불러온다.
- [ ] 블로그 생성에 사용할 특정 커밋들을 선택할 수 있다.
- [ ] 선택된 커밋의 상세 정보 또는 코드 변경 사항을 분석 대상으로 만든다.
- [ ] OpenAI 등 LLM을 활용해 커밋과 코드 변경 사항을 분석한다.
- [ ] 분석 결과를 사람이 읽기 좋은 개발 블로그 초안으로 변환한다.
- [ ] 초안에는 단순 커밋 나열이 아니라 개발 의도, 문제 해결 과정, 구현 내용, 결과가 드러나야 한다.

### 5.2 포스트 작성 및 편집 기능

- [ ] AI가 생성한 글을 사용자가 직접 수정할 수 있는 편집기 UI를 제공한다.
- [ ] 제목을 수정할 수 있다.
- [ ] 본문을 수정할 수 있다.
- [ ] 요약 또는 미리보기 문구를 수정하거나 자동 생성할 수 있다.
- [ ] 선택한 Repository, 브랜치, 커밋 정보가 글과 연결되어 있어야 한다.
- [ ] 저장 전 사용자가 생성 결과를 검토할 수 있어야 한다.
- [ ] 저장 중, 저장 완료, 저장 실패 상태를 구분한다.

### 5.3 저장된 포스트 보기 기능

- [ ] 저장된 글들을 카드 형태로 보여준다.
- [ ] 카드에는 제목이 표시된다.
- [ ] 카드에는 브랜치 태그가 표시된다.
- [ ] 카드에는 요약 미리보기가 표시된다.
- [ ] 카드에는 작성일 또는 저장일이 표시된다.
- [ ] 카드에는 관련 Repository 정보가 표시되면 좋다.
- [ ] 카드에는 발행 상태가 표시되면 좋다.
- [ ] 저장된 글을 다시 열어 편집할 수 있다.
- [ ] 최종 검토 후 블로그에 게시 또는 발행할 수 있다.

## 6. 프로그래밍 요구사항

### 6.1 AI 활용 구현

- [ ] AI를 단순 코드 생성 도구로만 쓰지 않고, 계획, 설계, 개발, 검증 workflow에 활용한다.
- [ ] 프로젝트를 진행하면서 나만의 AI workflow를 만든다.
- [ ] workflow는 한 번 작성하고 끝내지 않고 개발 중 계속 보완한다.
- [ ] 다음 주차까지 개선 가능한 방식으로 workflow를 남긴다.
- [ ] Agent 개발에 필요한 나만의 Skill을 하나 만들어 사용한다.
- [ ] Skill은 갑자기 억지로 만들기보다, 개발 중 발견한 좋은 process나 반복 패턴을 정리해 만든다.
- [ ] 만든 Skill이 어떤 상황에서 사용되는지 설명할 수 있어야 한다.

### 6.2 주차별 진행

- [ ] 전체 기능 범위를 2주간 진행할 수 있도록 나눈다.
- [ ] 각 주차별 목표를 스스로 정한다.
- [ ] 1주차에는 기획 구체화와 설계를 중심으로 진행할 수 있다.
- [ ] 2주차에는 설계 보완과 개발을 중심으로 진행할 수 있다.
- [ ] 프로젝트 저장소에 주차별 계획을 `.md` 파일로 작성한다.
- [ ] 주차별 계획 문서를 commit에 포함한다.
- [ ] 계획 문서에는 목표, 범위, 구현 항목, 검증 방법이 포함되어야 한다.

## 8. API 연동 체크리스트

### 8.1 GitHub API

- [ ] GitHub API 연동 방식에 대해 학습하고 적용한다.
- [ ] GitHub API 요청은 Express 기반 서버 환경에서 처리한다.
- [ ] 브라우저에서 GitHub token을 직접 노출하지 않는다.
- [ ] Repository 목록 API를 구현한다.
- [ ] Branch 목록 API를 구현한다.
- [ ] Commit 목록 API를 구현한다.
- [ ] Commit 상세 또는 diff 조회 API를 구현한다.
- [ ] API 실패 상태를 처리한다.
- [ ] GitHub rate limit 가능성을 고려한다.
- [ ] REST API 또는 GraphQL API 중 어떤 방식을 쓸지 결정하고 이유를 남긴다.

### 8.2 LLM API

- [ ] OpenAI 등 LLM API 연동 방식을 정한다.
- [ ] LLM API 요청은 서버에서 처리한다.
- [ ] 클라이언트에 LLM API key를 노출하지 않는다.
- [ ] 커밋 데이터를 LLM 입력에 맞게 정리한다.
- [ ] 너무 긴 diff를 그대로 보내지 않도록 요약 또는 제한 전략을 세운다.
- [ ] LLM 프롬프트에는 원하는 블로그 형식을 명확히 포함한다.
- [ ] LLM 응답 실패, 빈 응답, 형식 오류를 처리한다.
- [ ] LLM 결과를 사용자가 수정할 수 있는 초안으로 저장한다.

### 8.3 보안 및 환경 변수

- [ ] GitHub token은 `.env` 파일에서 관리한다.
- [ ] LLM API key는 `.env` 파일에서 관리한다.
- [ ] `.env`는 commit 대상에 포함하지 않는다.
- [ ] `.gitignore`에 `.env`가 포함되어 있는지 확인한다.
- [ ] 예시 환경 변수는 `.env.example`에 값 없이 남긴다.
- [ ] 토큰, key, 개인 정보가 코드, 문서, 로그, 최종 답변에 노출되지 않도록 한다.

## 9. AI 블로그 초안 품질 기준

- [ ] 초안은 단순 커밋 메시지 나열이 아니어야 한다.
- [ ] 어떤 문제를 해결했는지 드러나야 한다.
- [ ] 어떤 접근을 선택했는지 설명해야 한다.
- [ ] 주요 변경 파일 또는 코드 변경의 의미를 설명해야 한다.
- [ ] 구현 중 배운 점이나 다음 개선점을 포함하면 좋다.
- [ ] 개발 블로그 독자가 이해할 수 있는 문장으로 작성되어야 한다.
- [ ] 과장되거나 사실과 다른 내용이 생기지 않도록 원본 커밋 정보에 근거해야 한다.
- [ ] 사용자가 직접 다듬을 수 있도록 편집 가능한 초안으로 제공되어야 한다.

권장 초안 구조:

```md
# 제목
```

## 10. 테스트 및 검증 방식

- [ ] 구현 완료 조건을 미리 정의한다.
- [ ] Repository 목록이 정상적으로 표시되는지 확인한다.
- [ ] Branch 선택이 정상적으로 동작하는지 확인한다.
- [ ] Commit 목록이 정상적으로 표시되는지 확인한다.
- [ ] Commit 선택 상태가 정확히 유지되는지 확인한다.
- [ ] 선택한 Commit 기반으로 AI 초안이 생성되는지 확인한다.
- [ ] 생성된 초안을 사용자가 수정할 수 있는지 확인한다.
- [ ] 글 저장이 정상적으로 동작하는지 확인한다.
- [ ] 저장된 글이 카드 목록에 표시되는지 확인한다.
- [ ] 저장된 글을 다시 편집할 수 있는지 확인한다.
- [ ] 발행 상태 전환이 정상적으로 동작하는지 확인한다.
- [ ] API token이 클라이언트 번들에 포함되지 않았는지 확인한다.
- [ ] `.env`가 git에 포함되지 않았는지 확인한다.
- [ ] 네트워크 실패, 인증 실패, 빈 데이터 상태를 확인한다.
- [ ] LLM 응답 실패 시 사용자가 이해할 수 있는 메시지를 제공하는지 확인한다.

## 14. 앞으로 AI가 작업할 때 지켜야 할 기준

- [ ] 작업을 시작하기 전에 이 `checklist.md`를 먼저 확인한다.
- [ ] 새 기능을 만들 때 어떤 체크리스트 항목을 해결하는지 명시한다.
- [ ] 설계가 바뀌면 이 문서 또는 별도 계획 문서를 함께 업데이트한다.
- [ ] API key, token, `.env` 값은 절대 파일에 직접 쓰지 않는다.
- [ ] GitHub API 요청은 Express 서버를 통하도록 설계한다.
- [ ] LLM API 요청도 서버를 통하도록 설계한다.
- [ ] React는 사용자 흐름과 편집 경험을 중심으로 구현한다.
- [ ] 구현 후에는 테스트 또는 수동 검증 결과를 남긴다.
- [ ] 미션 요구사항과 다른 결정을 내릴 때는 이유를 문서화한다.
