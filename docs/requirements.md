# 🧠 스마트 블로그 기획서: AI 디펜스 튜터 (Smart Blog - AI Tutor)

## 1. 프로젝트 개요 (Project Overview)

### 1.1 프로젝트 배경 및 목적
단순히 코드를 요약해 주는 AI 서비스는 많지만, 이는 개발자의 성장을 오히려 방해할 수 있습니다. 본 프로젝트는 **"AI가 쓴 글을 내 블로그에 복사하는 것이 무슨 의미가 있을까?"**라는 고민에서 출발했습니다. 
스마트 블로그 서비스는 사용자의 GitHub 커밋 내역을 분석하여 단순히 요약 글을 던져주는 대신, 해당 코드의 '의도'와 '설계 방식'에 대해 깐깐하게 질문을 던집니다. 사용자는 이 질문에 대답하거나 피드백을 받는 과정을 통해 자연스럽게 학습(TIL, Today I Learned)을 진행하게 되며, 이 대화 내역이 훌륭한 회고록 형태의 블로그 초안으로 완성됩니다.

### 1.2 핵심 컨셉: "시니어 개발자와의 1:1 디펜스 면접"
- **단순 요약 지양:** 코드의 Diff(변경 사항)를 분석해 "어떤 로직을 추가했네요"라고 알려주는 것이 아니라, **"왜 이 로직을 선택했나요? 다른 대안은 없었나요?"**라고 묻는 튜터 페르소나를 부여합니다.
- **오답 노트의 자산화:** 주니어 개발자가 완벽한 대답을 하지 못하더라도 괜찮습니다. 대답을 포기하거나 힌트를 얻어 배운 내용 자체가 '오늘의 트러블슈팅 및 학습 기록'으로 변환되어 블로그 포스트로 저장됩니다.

### 1.3 기대 효과
- 사용자는 블로그 작성을 핑계로 자신이 작성한 코드를 다시 한번 복기하고, 부족한 CS/프레임워크 개념을 메울 수 있습니다.
- 백지상태에서 글을 쓰는 압박감 없이, AI와의 대화를 핑퐁하다 보면 자연스럽게 한 편의 기술 블로그 글이 완성됩니다.

## 2. 핵심 기능 및 UI 시나리오 (Core Features & UI Scenarios)

본 서비스는 크게 **[저장소 선택] ➡️ [AI 인터뷰] ➡️ [초안 검토 및 발행]**의 3단계 흐름으로 진행됩니다.

### 📍 Step 1: 시작 화면 (GitHub 저장소 및 커밋 선택)
사용자가 오늘 학습하고 리뷰할 코드를 선택하는 진입점입니다.

* **UI 레이아웃 (예상):**
  - 깔끔한 헤더 (서비스 로고 및 사용자 프로필)
  - 좌측/상단: 연동된 사용자의 GitHub Repository 목록 (드롭다운 또는 카드 리스트)
  - 우측/하단: 선택한 Repository의 최신 Commit 내역 리스트 (날짜, 커밋 메시지, 해시값 표시)
* **상세 기능 명세:**
  - **GitHub 연동:** 사용자의 Personal Access Token(PAT) 등을 활용해 GitHub API와 연동하여 리포지토리 목록을 Fetch 합니다.
  - **커밋 선택:** 특정 커밋을 클릭하면, 해당 커밋의 해시(Hash)값을 기반으로 Server에 코드 변경 사항(Diff) 데이터를 요청하며 **Step 2** 화면으로 전환됩니다.

---

### 📍 Step 2: AI 튜터 인터뷰 룸 (핵심 화면)
이 프로젝트의 메인 기능이자 가장 중요한 화면입니다. LLM이 코드 변경 사항을 분석해 사용자에게 질문을 던지고 답변을 유도합니다.

* **UI 레이아웃 (예상):**
  - **[상단] AI의 질문 영역:** LLM이 생성한 핵심 질문 1~2개가 채팅 말풍선이나 강조된 카드 형태로 표시됩니다. (예: *"이 커밋에서 `useEffect`의 의존성 배열을 비워두셨는데, 마운트 시점에만 API를 호출하도록 설계하신 구체적인 이유가 있나요?"*)
  - **[중단] 사용자 답변 영역:** 넓고 쾌적한 텍스트 에디터(`textarea`)가 배치되어 사용자가 자신의 생각이나 의도를 자유롭게 작성할 수 있습니다.
  - **[하단] 액션 버튼 (안전장치 포함):** 학습의 흐름이 끊기지 않도록 상황에 맞는 4가지 버튼을 제공합니다.
    1. `✅ 답변 제출`: 작성한 답을 AI에게 보내어 피드백을 요청합니다.
    2. `💡 힌트 보기`: 답변이 막힐 때 누르면, AI가 정답 대신 생각의 방향성을 잡아주는 짧은 힌트를 제공합니다.
    3. `🏳️ 모르겠어요 (해설 보기)`: 대답을 포기할 경우, AI가 상세한 정답과 개념 해설을 제공합니다. 이 해설은 곧바로 블로그 초안(오답 노트 형태)으로 저장됩니다.
    4. `⏭️ 질문 스킵 (일반 모드)`: 오늘은 학습할 에너지가 없을 경우, 인터뷰를 종료하고 해당 커밋에 대한 '일반적인 요약/릴리즈 노트' 형태의 블로그 초안을 즉시 생성합니다.

* **백그라운드 로직 (AI Workflow):**
  - Server는 전달받은 커밋 Diff를 LLM에게 보낼 때 강력한 프롬프트(System Prompt)를 적용합니다.
  - *프롬프트 예시:* "너는 10년 차 시니어 React 개발자야. 이 코드에서 주니어가 실수하기 쉽거나 개념적으로 깊게 이해해야 하는 부분에 대해 날카로운 질문 1개를 던져. 응답은 반드시 지정된 JSON 포맷(질문, 예상 정답, 힌트 포함)으로 줘."

---

### 📍 Step 3: 블로그 초안 검토, 수정 및 발행
AI와의 인터뷰(또는 스킵)가 종료된 후, 최종적으로 완성된 글을 다듬고 저장하는 공간입니다.

* **UI 레이아웃 (예상):**
  - **[좌측] 마크다운 에디터:** 노션(Notion)이나 벨로그(Velog)와 유사한 깔끔한 텍스트 에디터. AI가 조립해 준 초안이 마크다운 문법으로 렌더링 되어 있습니다.
  - **[우측] 최종 렌더링 뷰 (선택 사항):** 사용자가 에디터에서 글을 수정할 때 실시간으로 어떻게 보이는지 확인하는 프리뷰 화면.
  - **[우측 상단] 발행/저장 버튼:** 글을 최종 저장소(DB 또는 메모리)에 저장하는 버튼.
* **상세 기능 명세:**
  - **자동 초안 생성 로직:** Step 2에서의 결과를 바탕으로 글의 템플릿이 결정됩니다.
    - *답변 성공 시:* "오늘 구현한 내용 -> 나의 설계 의도 -> AI의 긍정적 피드백" 구조.
    - *모르겠어요 선택 시:* "오늘 구현한 내용 -> 내가 막혔던 부분 -> AI를 통해 새롭게 알게 된 CS 개념" 구조의 오답 노트.
    - *스킵 선택 시:* "이번 커밋의 주요 변경 사항 및 요약" 구조.
  - **포스트 관리:** 완료된 글은 '저장된 포스트 목록' 화면에서 카드 형태(제목, 생성 날짜, 썸네일 요약 포함)로 미리 보기가 가능하며, 언제든 다시 열어서 재수정할 수 있습니다.

## 3. 기술 스택 (Technology Stack)
- **Frontend (Client):** React, UI 라이브러리(Shadcn UI, Chakra UI 등 활용하여 디자인 코스트 최소화)
- **Backend (Server):** Express.js (GitHub API 및 LLM API 통신 중계 역할)
- **LLM API:** OpenAI API (GPT-4o-mini 등 경량 모델) 또는 Google Gemini API (비용 절감 목적)
