## 한 줄 정의

> 사용자의 GitHub 커밋을 LLM이 사람 친화적인 블로그 글로 바꿔주고, 사용자가 직접 다듬어 발행할 수 있는 서비스.

## 사용자

- **누구**: GitHub 에서 코딩하며 개발 블로그를 쓰고 싶지만 "글로 정리하는 일"이 귀찮은 개발자
- **핵심 문제**: 커밋은 매일 쌓이는데, 그 변화를 글로 풀어내려면 처음부터 다 쓰게 됨

## 핵심 기능 (F)

### F1. 저장소 연동 (Repository Link)
- F1.1 로그인된 GitHub 계정의 repository 리스트를 받아온다.
- F1.2 리스트는 검색 가능해야 한다.
- F1.3 카드/리스트에서 하나를 선택할 수 있다.

### F2. 브랜치·커밋 선택 (Branch & Commit Picker)
- F2.1 선택한 저장소의 브랜치 리스트를 받아온다.
- F2.2 선택한 브랜치의 최근 커밋 리스트(작성자, 메시지, 날짜, sha)를 받아온다.
- F2.3 커밋 카드에서 하나(또는 여러 개)를 선택할 수 있다.

### F3. AI 요약 (AI Summary)
- F3.1 선택된 커밋(들)의 메시지 + diff 를 LLM 에 전달한다.
- F3.2 LLM 응답을 "블로그 글 초안" 형식으로 받아 화면에 표시한다.
- F3.3 응답 생성 중에는 로딩 상태를 보여준다.
- F3.4 실패 시 에러 메시지와 재시도 버튼을 보여준다.

### F4. 편집기 (Editor)
- F4.1 AI 초안을 텍스트 영역에 채워 사용자가 수정 가능하게 한다.
- F4.2 글자 수가 보인다.
- F4.3 "저장" 클릭 시 in-memory + JSON 파일에 저장된다.

### F5. 카드형 포스트 목록 (Saved Posts)
- F5.1 저장된 포스트가 카드 형태로 보인다 (브랜치 태그, 요약 미리보기, 날짜).
- F5.2 카드에서 "수정하기" / "발행하기" 동작이 가능하다.
- F5.3 발행된 포스트는 시각적으로 구분된다.

### F6. 수정/발행 (Edit / Publish)
- F6.1 저장된 포스트를 다시 편집기로 열 수 있다.
- F6.2 발행 클릭 시 상태가 `draft` → `published` 로 바뀐다.

## 비기능 요구사항 (NF)

- **NF1 보안**: `GITHUB_TOKEN`, `OPENAI_API_KEY` 는 서버 사이드 `.env` 에만 존재. 클라이언트 번들에 포함되지 않아야 한다.
- **NF2 환경 분리**: 클라이언트는 Express 서버를 통해서만 외부 API 에 접근한다 (CORS / 토큰 노출 방지).
- **NF3 단순함**: 11~12주차 안에 끝낼 수 있는 범위. DB 미사용, 인증 미구현 (단일 사용자 가정).
- **NF4 응답성**: AI 요약을 제외한 모든 화면 전환은 200ms 안에 첫 반응이 보여야 한다.
- **NF5 회복성**: 외부 API (GitHub / OpenAI) 실패 시 사용자에게 명확한 메시지 + 재시도 경로.

## 가정 (Assumptions)

- 11~12주차 동안 사용자는 1명 (로컬에서 본인만 실행). 멀티 유저, 권한 모델, OAuth flow 는 범위 밖.
- GitHub 인증은 **Personal Access Token** (서버 사이드 `.env`) 한 개로 처리. OAuth 흐름은 12주차에도 미포함.
- 저장 매체는 in-memory 객체 + JSON 파일 영속화. SQLite/Postgres 는 도입 검토만 하고 12주차 결정 시점에 다시 본다.
- LLM 모델 기본값: `gpt-4o-mini` (비용/속도 균형).

## 명시적으로 범위 밖 (Out of scope)

- 다중 사용자, 로그인/회원가입 UI, OAuth callback
- 댓글, 좋아요, 구독, RSS
- 모바일 전용 디자인 (반응형은 최소한만)
- 발행된 포스트의 외부 호스팅(블로그 플랫폼 publish)
