# Implement Plan Item

`plans/weekN.md`에 적힌 work item 하나를 받아 **계획 → 구현 → 검증 → 커밋**까지
이어서 처리한다. 한 번에 한 항목만 다룬다. PR 생성은 범위 외 — 사용자가 직접 한다.

이 skill은 commit-to-blog 프로젝트 전용이다. spec/design 문서가 단일 source of truth이고,
plans/는 일정만 잡혀 있어 상세 내용이 없다는 전제 위에서 동작한다.

## 언제 트리거되는가

- "편집기 구현해줘", "포스트 저장 기능 만들자", "발행 기능 작업해" 처럼 plans/의 항목 하나를 가리키는 요청
- "이번주 작업 시작하자" → 사용자에게 어떤 항목인지 되묻고 하나로 좁힌 뒤 시작

여러 항목을 동시에 처리하려는 요청에는 사용하지 않는다 — 항목 하나로 좁힐 것을 권한다.
이미 코드가 있고 단순히 버그를 고치는 요청도 이 skill의 대상이 아니다.

## 사용하지 않는 경우

- 단순 버그 수정 / 리팩토링
- spec.md나 design.md를 *바꿔야 하는* 요청 (요구사항 변경은 사람의 결정 영역)
- 외부 블로그 발행 같이 spec에서 "범위 외"라고 명시된 작업

---

## Phase 1: 컨텍스트 로드 (메인이 직접)

다음 파일을 **반드시** 읽는다. 매번 새로 읽어야 한다 — 직전 세션에서 읽었더라도.

| 파일 | 역할 |
|------|------|
| `CLAUDE.md` | 프로젝트 전체 맥락·기술 스택·디렉토리 구조·제약 |
| `docs/spec.md` | 기능 정의, 아키텍처, 제약 |
| `docs/design.md` | 화면별 요구 구조 |
| `plans/weekN.md` | 어느 주차의 어떤 항목인지 확인 |

추가로:

1. 인자/대화에서 가리키는 항목을 `plans/` 파일 중 하나에서 정확히 매칭한다.
   - 매칭 후보가 2개 이상이거나 모호하면 사용자에게 어떤 항목인지 확인하고 진행한다.
   - 사용자가 plans/에 없는 작업을 요청하면 일단 *plans에 없는 작업입니다* 라고 알리고
     plans 항목으로 정리하길 원하는지 묻는다.
2. 해당 항목을 spec/design의 어느 섹션이 정의하는지 식별한다.
   - 예: "편집기 구현" → spec.md의 "1. 블로그 생성" 후반부 + design.md의 화면 2/3의 편집기 영역
   - spec/design에 매핑되는 내용이 없으면 사용자에게 *spec/design에 정의가 없습니다.
     spec/design을 먼저 보강해야 합니다* 라고 알리고 작업을 중단한다.
3. 현재 코드 상태를 파악한다.
   - `client/src/`, `server/src/` 디렉토리 구조와 관련 파일을 읽는다.
   - 이미 부분 구현이 있는지, 무엇이 비어있는지 확인한다.

### 1-A. Spec 공백 점검 — 추측 금지 게이트

Phase 1을 끝내기 전에 다음 항목을 **반드시 한 줄씩** 답해 본다. 답을 spec/design에서
*인용*할 수 없다면 그 자체가 공백이다. 공백이면 **추측하지 말고 사용자에게 물은 뒤** 답을
받고 진행한다. (구현을 시작하고 나서 "추측했다"고 보고하는 것은 늦었다.)

같이 점검할 표준 공백 항목 (이 프로젝트에서 반복적으로 빠지는 자리):

- **HTTP 메서드/경로/요청·응답 스키마**: spec/design에 명시 안 되어 있으면 적어도 사용자에게 *제안* 하고 OK를 받는다.
- **동일 동작 재호출의 정의**: "이미 발행된 포스트에 다시 발행 요청이 오면?" 같은 *상태 전이의 가장자리* 케이스. 보통 spec에 비어 있다. 가능한 두 답(예: idempotent 200 / 거절 409) 중 어느 쪽인지 사용자에게 묻는다.
- **에러 케이스의 HTTP 상태/사용자 표시**: 404/400/409/500 매핑이 design에 없으면 묻는다.
- **선행 plan 항목의 산출물에 의존하는 데이터**: 이 plan 항목이 "어떤 데이터/엔드포인트가 이미 있다고 가정"한다면 그게 진짜 있는지 코드로 확인. 없으면 1-B로 간다.

질문은 한 번에 묶어서 한다 (사용자 핑퐁 횟수 최소화). 답이 오면 Phase 2 분해에 그대로 반영한다.

### 1-B. 선행 미구현 plan 항목 처리

이 항목을 의미있게 동작·검증하려면 다른 plan 항목이 먼저 있어야 하는 경우가 있다.
(예: "발행 기능"은 "포스트 저장/조회"가 있어야 의미 있다.)

해당 항목이 아직 없거나 비어 있으면 **셋 중 하나로 선택**해 사용자에게 알리고 승인받는다:

1. **선행 항목을 먼저 진행**한다 — 이 항목 작업을 멈추고 선행 항목을 새 실행으로 처리한다. (가장 정석)
2. **최소 stub만 같이 만든다** — 이 항목 검증에 꼭 필요한 최소 데이터/엔드포인트만 같이 만들고, 선행 항목의 본 작업은 *그 항목 실행 시* 확장하기로 한다. 보고서에 "이 부분은 X plan 항목 책임이며, 다음 실행에서 확장될 예정"이라고 명시.
3. **선행 없이 진행** — 진짜로 독립이 가능하다면. 보통은 아니다.

추측해서 "여기까지 한 김에 다 만들겠다" 라고 스코프를 부풀리지 않는다. 작게 잘라야 검토 비용이 낮다.

## Phase 2: 구현 계획 (메인이 직접)

읽은 내용을 토대로 항목 하나를 작은 단위로 분해한다.

### 분해 원칙 — **병렬 가능성을 먼저 본다**

서브에이전트는 **동시에 여러 개를 돌릴 수 있을 때만 시간을 절약**한다. 분해할 때는 항상
"이 두 태스크는 같은 파일을 건드리는가?", "한쪽 결과가 다른 쪽 입력인가?"를 먼저 묻는다.
답이 모두 "아니오"라면 두 태스크는 **반드시 병렬로 돌린다**.

축은 다음을 기본으로 본다:

- **server 변경**: Express 라우트·서비스·DB 접근 — 새 파일이면 다른 server 작업과 병렬 가능
- **client 변경**: React 페이지·컴포넌트·API 호출 — 새 파일이면 다른 client 작업과 병렬 가능
- **테스트**: 같은 PR에서 같이 만든다 (아래 "테스트 포함" 참고)
- **공유 타입**: 양쪽 모두 의존하므로 보통 가장 먼저 직렬로 처리

### 직렬을 강제하는 조건

다음에 해당하면 직렬로 묶는다 (`addBlockedBy` 설정):

- **공유 타입을 먼저 정의해야 하는 경우**: client/server 어느 쪽이든 같은 request/response 모양을 쓴다면 타입을 먼저 만들고 양쪽이 import.
- **client가 server API를 호출하는 경우**: client 통합 테스트가 실제 라우트 모양에 의존하므로, 라우트 시그니처가 확정될 때까지 client API 호출 코드/테스트는 시작하지 않는다.
- **DB 스키마 변경이 라우트 로직에 선행되어야 하는 경우**.

이외의 경우는 모두 병렬이 기본이다. 예를 들어 "server에 새 라우트 A 추가" + "client에 무관한 페이지 B의 빈 라우팅 추가"는 병렬.

### 테스트도 분해 단위에 포함한다

이 프로젝트에는 아직 테스트가 없다. plan 항목을 구현할 때 **그 항목과 직접 관련된 자동
테스트를 함께 추가**한다. 테스트 도구가 아직 없으면 첫 항목에서 같이 세팅한다:

| 영역 | 도구 | 위치 |
|------|------|------|
| server | `vitest` + `supertest` (HTTP 라우트) | `server/src/**/*.test.ts` |
| client | `vitest` + `@testing-library/react` | `client/src/**/*.test.tsx` |

테스트 도구가 `package.json`에 없으면 Phase 2 분해의 **첫 번째 서브태스크**가 "테스트 세팅"이 된다. 이 서브태스크는 다른 모든 태스크의 선행이다.

테스트의 범위는 그 plan 항목의 **유저-가시 동작**에 한정한다. 단위 100% 커버리지가 아니라
"라우트가 의도한 응답을 주는가", "컴포넌트가 사용자 동작에 의도한 결과를 내는가" 수준.
구현 코드와 테스트는 **같은 서브태스크 안에서 같은 subagent가 함께 작성**한다 (분리하면
구현자가 테스트하기 쉬운 형태로 코드를 짜는 이점을 잃는다).

### 사용자에게 공유

다음을 보여주고 사용자 승인을 받는다:

```
대상 항목: <plan 항목 이름> (plans/weekN.md)
근거 spec: <spec.md의 어느 섹션>
근거 design: <design.md의 어느 화면>
서브태스크 (병렬/직렬 표시):
  1. [server] 라우트 X 추가 + 라우트 테스트         (병렬 가능: 2와)
  2. [client] 페이지 Y 추가 + 컴포넌트 테스트       (병렬 가능: 1과)
  3. [client] Y에서 X 호출하는 통합 부분            (직렬: 1, 2 이후)
검증 계획: tsc / lint / build / 자동 테스트 / <어떤 수동 검증>
```

승인 후 Phase 3으로 간다. 사용자가 분해를 바꾸길 원하면 반영하고 다시 보여준다.

각 서브태스크는 `TaskCreate`로 등록하고 의존관계가 있으면 `addBlockedBy`로 표시한다.

## Phase 3: 구현 (subagent 위임)

각 서브태스크를 subagent에 위임한다. 메인이 직접 코드를 쓰지 않는다.

### 병렬 실행 — 메시지 1개에 Agent 호출 N개

`blockedBy`가 없는 태스크들은 **반드시 같은 메시지 안에 여러 Agent 호출을 묶어** 동시에
실행한다. 한 번에 하나씩 보내면 직렬화돼 시간이 곱절로 든다. 예를 들어 서브태스크 1, 2가
서로 독립이면:

```
한 응답 안에:
  Agent({description: "...", prompt: "<태스크 1>"})
  Agent({description: "...", prompt: "<태스크 2>"})
```

`blockedBy`가 걸린 태스크는 선행 태스크가 Phase 4를 통과한 뒤에야 시작한다.
선행이 끝나는 즉시 후행의 입력(라우트 시그니처, 새 타입 등)을 정리해서 다음 subagent에 넘긴다.

### subagent 프롬프트에 반드시 포함 — 정확도용 게이트

subagent는 메인의 대화 컨텍스트를 보지 못한다. 모호하게 던지면 멋대로 추측한다.
다음을 빠짐없이 넣어 추측 공간을 좁힌다:

```
## 프로젝트 루트
<절대 경로>

## 대상 항목
<plan 항목 이름>과 <spec/design의 어느 섹션이 근거인지 — 인용 문장 포함>

## 입력/출력 계약 (Phase 1-A에서 확정됨)
- HTTP 라우트: METHOD /path, request body 스키마, response 스키마
- 컴포넌트: props 시그니처, 트리거 이벤트, 호출하는 API
- 함수: 시그니처, 에러 케이스
- 가장자리 케이스(중복 호출, 이미 같은 상태 등)의 정의된 응답
**여기 적힌 모든 항목은 메인이 사용자에게 확인 받은 결정이다.** 구현하면서 새로운 모호함이 생기면
혼자 결론 내지 말고 작업을 멈추고 메인에게 보고하라.

## 참고 코드 (반드시 먼저 읽어라)
- <기존 비슷한 라우트 파일 경로> ← 같은 패턴을 따른다
- <기존 비슷한 컴포넌트 경로>
- server/src/index.ts ← Express 부트스트랩, 미들웨어 순서

## 구현 내용
<수정/생성할 파일 경로와 무엇을 구현할지>

## 테스트도 함께
- 작성할 테스트 파일: <경로>
- 검증할 동작: <라우트가 200/4xx를 언제 주는지, 컴포넌트가 클릭 시 무엇을 호출하는지>

## 제약 (CLAUDE.md, 어겨선 안 됨)
- GitHub / LLM API 호출은 server에서만. client에 토큰을 두지 말 것.
- 환경변수는 server/.env. 새 변수 추가 시 CLAUDE.md의 .env 예시도 업데이트.
- 외부 블로그 publishing은 범위 외 — status 전환까지만.

## 완료 조건 (체크 후 보고)
- [ ] 위 "참고 코드"를 실제로 읽었다 (Read 호출 후 진행)
- [ ] 수정 전 대상 파일의 현재 내용을 읽었다
- [ ] server 수정 시: 라우트는 zod로 body 검증
- [ ] server 수정 시: GitHub/LLM 호출은 @octokit/rest / @anthropic-ai/sdk 사용
- [ ] client 수정 시: 토큰을 다루는 코드를 추가하지 않았다
- [ ] 테스트 파일을 작성했고 로컬에서 통과 (npx vitest run <path>)
- [ ] tsc 통과 (npx tsc --noEmit) — type-only 변경도 반드시 실행
- [ ] eslint 통과 (npx eslint <변경 파일>)
- [ ] 새 의존성은 해당 workspace의 package.json에만 추가
- [ ] (해당 시) 새 env 변수를 CLAUDE.md의 server/.env 예시에 반영

## 보고할 것
- 수정/생성한 파일 절대 경로 목록
- 추가한 의존성 (있다면)
- 위 체크리스트의 각 항목 통과 여부 (단순 "OK"가 아니라 실행 결과 한두 줄)
```

서브에이전트 시작 직전에 `TaskUpdate(status: in_progress)`, Phase 4 통과 후 `completed` 처리.

### 메인은 보고를 그대로 믿지 않는다

subagent가 "통과했다"고 보고해도 Phase 4에서 메인이 직접 다시 돌린다. subagent에 hook이 적용되지 않으므로 *진짜로* 돌았는지 모른다. Phase 4의 의미는 여기에 있다.

## Phase 4: 검증 (메인이 직접)

서브에이전트의 보고를 그대로 믿지 않는다. 메인이 직접 명령을 돌려 결과를 본다.
4-1 ~ 4-5 중 변경된 영역에 해당하는 항목을 **모두** 통과해야 한다.

### 4-1. 타입체크

```bash
# server 변경이 있던 경우
cd <프로젝트루트>/server && npx tsc --noEmit

# client 변경이 있던 경우
cd <프로젝트루트>/client && npx tsc --noEmit
```

### 4-2. Lint

```bash
# 변경된 workspace에서
npx eslint <변경된 파일들>
```

루트에 lint 스크립트가 정의돼 있으면 그쪽을 우선 사용한다 (`npm run lint -w client` 등).
경로/스크립트가 명확하지 않으면 `package.json`을 읽어 정확한 명령을 찾는다.

### 4-3. 빌드

```bash
# client 변경이 있던 경우
cd <프로젝트루트>/client && npm run build

# server 변경이 있던 경우 (tsc 빌드)
cd <프로젝트루트>/server && npm run build
```

빌드 스크립트가 없으면 추가하지 말고 4-1의 tsc로 갈음한다.

### 4-4. 자동 테스트 ← 반드시 실행

이 단계는 **subagent가 새로 추가한 테스트 + 기존 테스트 전체**를 메인이 직접 돌린다.

```bash
# server 변경이 있던 경우
cd <프로젝트루트>/server && npx vitest run

# client 변경이 있던 경우
cd <프로젝트루트>/client && npx vitest run
```

- `vitest run`은 watch 모드를 끄는 1회성 실행. `npx vitest`만 치면 안 된다 (watch가 켜져 멈춤).
- 실패 케이스가 새로 생긴 코드와 직결되면 4-6으로 간다.
- 기존 테스트가 깨졌다면 회귀이므로 같은 PR에서 잡는다.
- 새 테스트가 너무 적다 (예: 라우트 추가했는데 라우트 테스트 0개) 싶으면 부족분을 같은 sub-task에 다시 위임한다.

### 4-5. 수동 동작 검증 — 자동으로 닿는 데까지만

자동으로 가능한 만큼 한다. 사람 눈이 필요한 부분은 정직하게 "이건 브라우저에서 봐달라"고 명시한다.

**server-only 변경 (라우트 추가 등):**

1. `cd server && npm run dev`를 `run_in_background`로 띄운다
2. `Monitor`로 "listening on" 로그를 기다린 뒤 `curl`로 새 엔드포인트를 호출한다
3. 응답 상태/스키마가 spec에서 의도한 모양인지 확인 (4-4의 supertest가 이미 한 검증과 중복돼도 dev 서버가 실제로 뜨는지 확인하는 의미가 있음)
4. 끝나면 background 프로세스를 종료

**client 변경 (UI 페이지/컴포넌트):**

- `cd client && npm run dev`를 background 실행하고 부팅 로그(에러/경고)만 확인한다.
- UI 상호작용 자체의 정확성은 자동 검증 범위 밖이다.
  → 사용자에게 어떤 URL로 들어가 무엇을 클릭/입력해 확인하면 되는지 **체크리스트로 알린다**.
- "UI를 봤다"고 추측해서 보고하지 않는다. 보지 못한 것은 보지 못했다고 말한다.

### 4-6. 실패 시

`SendMessage`로 해당 subagent에게 실패한 검증 항목과 원인을 전달, 수정 후 Phase 4를 재실행.
2회 재시도해도 통과하지 못하면 사용자에게 알리고 멈춘다.

## Phase 5: 커밋

검증이 모두 통과한 후에만 커밋한다.

### 5-1. 커밋 메시지 스타일 확인

`git log --oneline -n 10`으로 최근 커밋 메시지 톤을 확인한 뒤 그 스타일을 따른다.
현재 레포는 짧은 한 줄(영문 명령형) 스타일.

### 5-2. 커밋

```bash
git add <변경된 파일들>
git commit -m "$(cat <<'EOF'
<짧은 한 줄 요약>

Co-Authored-By: Claude Opus 4.7 (1M context) <noreply@anthropic.com>
EOF
)"
```

- 한 항목 = **한 커밋**을 기본으로 한다. server와 client 변경이 한 기능이면 같이 묶는다.
- 서브태스크 단위로 잘게 쪼개지 않는다 (검토 부담만 늘어남).

### 5-3. PR 생성하지 않음

PR은 사용자가 직접 만든다. 대신 결과 보고에 `gh pr create` 예시 한 줄을 넣어준다.

## 메모

- **Skill은 사람을 대체하지 않는다.** spec/design이 모호하면 추측해서 채우지 말고 사용자에게 묻는다.
- 검증 단계에서 "통과한 것만 통과했다고 보고한다." 자동 검증 영역과 사람이 봐야 하는 영역을 흐리지 않는다.
- 한 번 실행에 처리하는 plan 항목은 1개. 욕심내지 않는다.
