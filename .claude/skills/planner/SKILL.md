# /planner

## 실행 순서

### 1. 컨텍스트 로드
- `@progress/current.md` 읽기
- current.md의 "관련 docs" 항목의 파일 로드
- current.md의 "관련 rules" 항목의 파일 로드
- 관련 코드 파일이 있다면 확인

### 2. 문제 정의

분석 후 문제를 정의하고 확인받는다:

```
─────────────────────────
CP1: 문제 정의 확인
─────────────────────────
Task: [task명]

무엇을 만드는가:
[설명]

왜 만드는가:
[이유]

핵심 제약사항:
- ...
─────────────────────────
→ 이 정의가 맞나요? 빠진 부분이 있나요?
```

### 3. 접근 방법 협의

선택지를 제시하고 방향을 함께 결정한다:

```
─────────────────────────
CP2: 접근 방법 선택
─────────────────────────
방법 A: [설명] ← 추천
  장점: ...
  단점: ...

방법 B: [설명]
  장점: ...
  단점: ...
─────────────────────────
→ 어떤 방향으로 할까요?
```

### 4. 실행 계획 초안 작성 후 확인

```
─────────────────────────
CP3: 실행 계획 확인
─────────────────────────
실행 단계 (총 N개):
- [ ] 1. ...
- [ ] 2. ...
- [ ] 3. ...

예상 변경 파일:
- ...

리스크:
- ...
─────────────────────────
→ 이 계획에서 수정하거나 추가할 게 있나요?
```

### 5. plan.md 저장 + 커밋

```
─────────────────────────
CP4: 최종 확인
─────────────────────────
→ progress/plan.md 에 저장할까요?
```

저장 완료 후 커밋을 진행한다:
```
git add progress/plan.md progress/current.md
git commit -m "workflow/plan: [task명]"
```

커밋 완료 후:
```
─────────────────────────
커밋 완료
─────────────────────────
→ /executor 로 넘어갈까요?
```
