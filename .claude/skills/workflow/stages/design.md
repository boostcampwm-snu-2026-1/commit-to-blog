# 설계 단계

## 목적
구현에 들어가기 전에 무엇을 어떻게 만들지 사용자와 합의한다. 이 단계가 탄탄할수록 구현이 빠르고 수정이 줄어든다.

## 절차

### 1. checklist.md 읽기
프로젝트 루트의 `checklist.md`를 읽고 `- [ ]`로 표시된 항목 중 가장 위에 있는 것을 선택한다.

checklist.md가 없으면: "프로젝트 루트에 checklist.md가 없어요. 만들어서 작업할 항목을 추가해주세요." 라고 안내하고 종료한다.

미완료 항목이 없으면: "checklist.md의 모든 항목이 완료되었어요!" 라고 안내하고 종료한다.

### 2. 항목 검토
선택한 항목이 너무 크거나 모호하다면, 더 작은 단위로 나눌 것을 제안한다.

예: "이 항목은 범위가 넓어서 한 번에 하기 어려울 것 같아요. 이렇게 나눠보는 건 어떨까요?" 하고 세분화 예시를 제시한다. 사용자가 동의하면 checklist.md를 직접 수정하거나 수정을 부탁한 뒤 다시 시작한다.

### 3. 구현 방법 인터뷰
항목이 확정되면 사용자가 생각하는 구현 방향을 먼저 들어본다. 한 번에 모두 묻지 말고 자연스럽게 대화하며 아래 내용을 파악한다.

- 어떤 파일을 수정/생성할 것인지
- 어떤 방식으로 구현할 것인지 (라이브러리, 패턴, 접근법 등)
- 이미 알고 있는 제약 조건이나 요구사항

### 4. 비판적 검토 및 개념 제시
사용자의 설계 방향을 파악했으면, 그 방식에서 발생할 수 있는 문제나 더 나은 방법을 능동적으로 찾아낸다.

단순히 문제를 지적하는 게 아니라, **관련 개념을 짧게 소개한 뒤 이 상황에서 어떻게 적용하면 좋을지 사용자가 스스로 생각해보게 질문**한다. 사용자가 개념을 배우면서 설계를 발전시킬 수 있도록 돕는 게 목적이다.

검토할 항목:
- **오류 가능성**: 예외나 버그가 발생할 수 있는 상황
- **비효율**: 더 간단하거나 성능이 좋은 방법이 있는 경우
- **누락된 케이스**: 사용자가 언급하지 않았지만 처리해야 할 상황

질문 형식:
```
[개념 이름]이라는 게 있는데, [한 줄 설명]이에요.
이 부분에서는 어떻게 적용하면 좋을 것 같아요?
```

예시:
- "옵셔널 체이닝이라는 게 있는데, 중첩된 객체에서 값이 없을 때 에러 대신 undefined를 반환해줘요. 이 데이터를 받아오는 부분에서 어떻게 활용할 수 있을 것 같아요?"
- "디바운스라는 개념이 있는데, 연속으로 발생하는 이벤트를 일정 시간 후 한 번만 처리하는 거예요. 이 검색 기능에 적용하면 어떤 점이 좋아질 것 같아요?"

문제가 없으면 질문 없이 넘어가도 된다. 억지로 개념을 끼워 넣을 필요는 없다.

### 5. 설계 요약 제시
인터뷰가 끝나면 합의된 내용을 아래 형식으로 요약해서 보여준다.

```
## 설계 요약
- 항목: [checklist 항목명]
- 수정/생성할 파일: [파일 목록]
- 구현 방식: [합의된 접근법]
- 주의사항: [엣지 케이스, 제약 조건]
```

사용자가 요약을 확인하고 문제가 없으면 구현 단계로 넘어간다.

## 완료 조건
사용자가 설계 요약을 확인하고 진행에 동의했을 때 → `stages/implement.md`로 이동
