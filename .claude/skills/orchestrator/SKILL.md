# Orchestrator

## 내가 알고 있는 것

### 프로젝트 시스템 구조
- **KB** (`docs/`): 도메인 지식, 아키텍처 결정. 코드 구조를 미러링한다.
- **Rules** (`rules/`): 코딩/디자인/협업 규칙. 필요한 작업 시 선택적으로 로드한다.
- **Progress** (`progress/`): 진행 상황. 세션 간 맥락을 유지한다.

### 워크플로우 스킬
| 스킬 | 역할 | 핵심 결과물 |
|------|------|------------|
| `/whattodo-next` | backlog에서 task 선택 | `progress/current.md` |
| `/planner` | 문제 정의 + 계획 수립 | `progress/plan.md` |
| `/executor` | 계획 실행 | 코드 변경 + `progress/execution-log.md` |
| `/evaluator` | 검증 + 완료 처리 | `progress/done.md` 이동 |
| `/summary` | 사이클 요약 | 완료 후 핵심 내용 출력 |

### 협업 원칙 (@rules/workflow.md)
- 가정하지 말고 질문한다
- 사용자 확인 없이 단계를 넘어가지 않는다
- 모호한 것은 즉시 멈추고 보고한다

## 세션 시작 루틴

아래 순서로 현재 상태를 파악한다:

1. `@progress/current.md` 읽기 → 진행 중인 task 확인
2. `@progress/plan.md` 읽기 → 이어받을 계획 있는지 확인
3. 상태에 따라 브리핑 후 다음 액션 제안

### 상태별 대응

**A. 진행 중인 task + plan 있음**
```
─────────────────────────
세션 브리핑
task: [task명]
현재 단계: executor 대기 중 (plan.md 있음)
─────────────────────────
→ /executor 로 이어서 진행할까요?
```

**B. 진행 중인 task 있으나 plan 없음**
```
─────────────────────────
세션 브리핑
task: [task명]
현재 단계: planner 필요
─────────────────────────
→ /planner 로 계획을 수립할까요?
```

**C. 진행 중인 task 없음**
```
─────────────────────────
세션 브리핑
현재 진행 중인 task 없음
─────────────────────────
→ /whattodo-next 로 새 task를 선택할까요?
```

## 사용자 요청 라우팅

세션 중 사용자 요청을 받으면 적절한 스킬로 안내한다.

| 요청 유형 | 안내 |
|----------|------|
| "뭐 하지?" / "다음 뭐해?" | `/whattodo-next` |
| "계획 짜줘" / "어떻게 만들지?" | `/planner` |
| "만들어줘" / "구현해줘" | `/executor` (plan.md 확인 후) |
| "검토해줘" / "완료야" | `/evaluator` |
| "요약해줘" / "뭐 했지?" | `/summary` |
| "문서 어디 있어?" | `@docs/README.md` 로드 후 안내 |
