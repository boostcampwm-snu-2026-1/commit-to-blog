# CLAUDE.md — AI 활용 워크플로우

이 파일은 commit-to-blog 프로젝트에서 Claude Code를 어떻게 활용했는지 기록합니다.

---

## AI 활용 전략

### 역할 분담

| 역할 | 담당 |
|------|------|
| 기능 범위 결정, 우선순위 | 사람 |
| 설계 문서 작성 (PLAN.md) | Claude (초안) → 사람 (검토/확정) |
| 코드 구현 | Claude |
| 동작 확인, 커밋 메시지 작성 | 사람 |
| 디버깅 방향 판단 | 사람 → Claude (실행) |

### 기본 원칙

- **feature 단위로 요청** — 한 번에 하나의 기능만 구현 요청. 완료 후 직접 확인하고 커밋.
- **설계 먼저, 구현 나중** — PLAN.md를 먼저 작성해 전체 구조를 확인한 뒤 구현 진입.
- **커밋 메시지는 직접 작성** — Claude가 초안을 제안하면 본인이 이해한 내용 기반으로 수정.

---

## 워크플로우

```
1. 기능 요구사항 분석 (사람)
        ↓
2. PLAN.md로 설계 확정 (Claude 초안 → 사람 검토)
        ↓
3. "feat N 구현해줘" 요청 (사람)
        ↓
4. Claude가 코드 작성 + 동작 테스트
        ↓
5. 브라우저/curl로 직접 동작 확인 (사람)
        ↓
6. "커밋 메시지 작성해줘" 요청 → 확인내용/이해 안 된 부분 채워서 커밋 (사람)
        ↓
7. 다음 feature로 반복
```

---

## 커밋 메시지 형식

```
feat: #N 기능명
- 확인내용: 직접 확인한 동작 내용
- 이해 안 됐던 부분: 처음엔 몰랐지만 이번에 알게 된 것
```

---

## 효과적이었던 질문 패턴

| 상황 | 질문 방식 |
|------|----------|
| 기능 구현 | "feat N 구현하고 커밋 메시지 알려줘" |
| 에러 발생 | 에러 메시지 그대로 붙여넣기 |
| 설계 고민 | "A와 B 중 어떤 게 더 적합해?" |
| 개념 이해 | "왜 이렇게 구현했어?" |

---

## 주의했던 점

- Claude가 생성한 코드는 반드시 직접 실행해서 확인
- 이해 안 되는 코드는 그냥 넘기지 않고 질문
- 설계 결정(스택 선택, DB 구조 등)은 Claude 제안을 참고하되 직접 판단
