# Implementation Checklist

## MVP 목표

- [ ] GitHub 계정으로 로그인할 수 있다.
- [ ] 사용자가 저장소, 브랜치, 커밋을 선택할 수 있다.
- [ ] 선택한 커밋을 바탕으로 AI가 Markdown 블로그 초안을 생성한다.
- [ ] 사용자가 초안을 편집하고 저장할 수 있다.
- [ ] 저장된 글을 발행하면 공개 URL로 비로그인 접근이 가능하다.

## 0. 기반 정리

- [ ] 프로젝트 구조와 라우트 그룹이 기능별로 정리되어 있다.
- [ ] Prisma schema와 migration이 현재 데이터 모델을 반영한다.
- [ ] 공통 서버 유틸, UI 유틸, 테스트 인프라가 유지 가능한 위치에 있다.
- [ ] formatting, linting, typecheck, test 명령이 로컬에서 일관되게 동작한다.

## 1. 인증과 세션

- [ ] GitHub OAuth 로그인이 동작한다.
- [ ] 사용자, 계정, 세션 정보가 DB에 저장된다.
- [ ] GitHub access token을 서버에서 사용할 수 있게 보관한다.
- [ ] 인증이 필요한 화면은 미로그인 사용자를 로그인 페이지로 보낸다.
- [ ] 헤더에서 사용자 정보와 로그아웃 액션을 제공한다.

## 2. GitHub 데이터 연동

- [ ] 로그인한 사용자의 public repository 목록을 가져온다.
- [ ] 선택한 repository의 branch 목록을 가져온다.
- [ ] 선택한 branch의 commit 목록을 가져온다.
- [ ] API 입력 검증과 인증 실패 처리가 일관되게 동작한다.
- [ ] GitHub rate limit 또는 외부 API 실패를 사용자에게 명확히 전달한다.

## 3. 블로그 생성 플로우

- [ ] 생성 화면에서 repository를 검색하고 선택할 수 있다.
- [ ] repository 선택 후 branch를 선택할 수 있다.
- [ ] branch 선택 후 commit을 여러 개 선택할 수 있다.
- [ ] 선택 상태가 AI 요약 요청에 필요한 형태로 정리된다.
- [ ] 로딩, 빈 결과, 실패 상태가 화면에서 구분된다.

## 4. AI 초안 생성

- [ ] 선택한 commit의 메시지, 파일 변경 목록, 통계를 AI 입력으로 구성한다.
- [ ] OpenAI 호출은 서버에서만 수행된다.
- [ ] 생성 결과가 Markdown 초안으로 스트리밍된다.
- [ ] 큰 입력은 토큰 한도를 넘지 않도록 압축하거나 제한한다.
- [ ] 생성된 초안을 draft post로 저장할 수 있다.
- [ ] AI 모델과 토큰 사용량 같은 메타데이터를 기록한다.

## 5. Markdown 편집기

- [ ] 저장된 draft를 편집 화면에서 불러온다.
- [ ] 제목과 본문을 수정할 수 있다.
- [ ] Markdown preview가 본문과 함께 제공된다.
- [ ] 코드 블록 하이라이팅이 동작한다.
- [ ] 편집 내용이 자동 저장되거나 명확한 저장 액션으로 반영된다.
- [ ] 다른 사용자의 글에는 접근할 수 없다.

## 6. 저장 글 관리와 발행

- [ ] 저장된 글 목록을 최신 수정순으로 보여준다.
- [ ] 글 카드에서 제목, 요약, 상태, 출처 정보를 확인할 수 있다.
- [ ] draft 글을 published 상태로 전환할 수 있다.
- [ ] 발행 시 고유한 slug와 공개 URL을 생성한다.
- [ ] 발행된 글도 다시 편집할 수 있다.
- [ ] 글 삭제는 확인 절차를 거친다.

## 7. 공개 블로그 페이지

- [ ] 발행된 글은 `/blog/[slug]`에서 비로그인으로 접근할 수 있다.
- [ ] draft 글은 공개 URL에서 노출되지 않는다.
- [ ] 공개 페이지는 편집기와 같은 Markdown 렌더링 규칙을 사용한다.
- [ ] 제목, 설명, Open Graph 등 기본 메타데이터를 제공한다.

## 8. 설정과 계정 정보

- [ ] 설정 화면에서 GitHub 연결 상태를 확인할 수 있다.
- [ ] 사용자 프로필 정보를 표시한다.
- [ ] 누적 AI 사용량을 확인할 수 있다.
- [ ] 로그아웃이 정상 동작한다.
- [ ] 필요 시 GitHub 재연결 흐름을 제공한다.

## 9. 운영 준비

- [ ] 필수 환경 변수를 검증하고 누락 시 빌드가 실패한다.
- [ ] 로컬, 테스트, 프로덕션 환경 변수를 구분한다.
- [ ] CI에서 typecheck, lint, unit test가 실행된다.
- [ ] E2E 테스트는 테스트용 인증/DB 전략을 사용한다.
- [ ] Vercel 배포 환경에서 로그인부터 공개 페이지까지 동작한다.

## 품질 기준

- [ ] API는 인증, 입력 검증, 소유권 검사를 일관되게 수행한다.
- [ ] 외부 API 호출은 테스트에서 mock 처리한다.
- [ ] 사용자에게 보이는 실패 상태는 복구 가능한 메시지로 표현한다.
- [ ] 서버 전용 secret은 클라이언트 번들로 노출되지 않는다.
- [ ] MVP 완료 시나리오가 E2E로 검증된다.

## MVP 완료 조건

- [ ] 로그인한다.
- [ ] repository, branch, commit을 선택한다.
- [ ] AI로 블로그 초안을 생성한다.
- [ ] 초안을 편집하고 저장한다.
- [ ] 글을 발행한다.
- [ ] 비로그인 브라우저에서 공개 URL을 열어 본문을 확인한다.
